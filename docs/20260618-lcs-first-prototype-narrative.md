## Problem Statement

The current prototype presents diagnosis, learning paths, practice, search, and architecture as parallel capabilities. For a support-program reviewer or mentor, that breadth can obscure the product's central differentiator: Learning Context Snapshot (LCS), which lets an AI mentor understand the learner's current content, progress, and recent practice failure before answering.

The prototype should tell one complete story: a bootcamp graduate fails a practical task, sees the limit of a generic AI answer, receives a context-aware DevPath review through LCS, and gets the next learning task adjusted as a result.

## Demand Evidence

Document 29 grounds the problem in the founder's own developer experience and identifies bootcamp graduates, students, and junior developers without mentors as intended users. It also records selection for `모두의 창업 1기` and an assigned mentor. This design document makes no additional demand claims beyond that source.

## Status Quo

The product framing in document 29 contrasts three disconnected workflows:

- Courses, books, and video content provide material but do not continuously adapt the next task to a learner's recent failure.
- Generic AI tools can answer a pasted question, but the learner must manually reconstruct learning history, current progress, and the error context.
- Bootcamp support is intensive while the program is running, but a graduate can lose that context after completion while preparing for employment.

## Target User & Narrowest Wedge

The target user is a bootcamp graduate preparing for employment whose immediate fear is not getting hired.

The narrowest prototype wedge is one failed backend practice task. The user should be able to see, in a single flow, why the generic answer is insufficient, what LCS attaches automatically, how the AI review changes because of that context, and how the learning path changes afterward.

## Constraints

- Keep development moving; this design changes the prototype's narrative priority rather than creating a stop/go gate.
- Reuse the existing Flutter Web prototype and its diagnosis, path, sandbox, and review simulations.
- Keep the prototype static: no production API, authentication, server persistence, or live AI dependency is required.
- Make the core story understandable in a three-minute reviewer demo.
- Keep learner-facing copy in Korean and preserve desktop and mobile usability.
- Continue distribution through the existing GitHub Pages deployment path.

## Premises

1. The prototype is a tool for persuading reviewers and mentors of the development direction, not a reason to stop development.
2. The first-screen protagonist is an AI mentor that knows the learner's context, not the platform's full feature structure.
3. Diagnosis, path, and sandbox remain in the product, but each supports one LCS before-and-after story.

## Approaches Considered

### Approach A: Emphasize LCS Within the Current Structure

Keep the current navigation and feature order, but strengthen LCS labels and explanatory copy. This has the smallest change surface, but still asks the reviewer to assemble the product story from separate screens.

### Approach B: One Closed Learning Loop

Reorder the prototype around `practice failure -> context auto-attach -> AI review -> path correction`. This makes the differentiator causal and visible: the path changes because the mentor understood the learner's context.

### Approach C: Generic AI vs. DevPath Comparison

Place a generic answer and an LCS-informed answer side by side. This makes the contrast immediate, but comparison alone does not prove that the answer changes the learner's next action.

## Recommended Approach

Use Approach B as the structure, with the strongest element of Approach C embedded in it, and begin with the practice failure.

The approved sequence is:

1. **Practice failure:** Show the failed task and the learner's uncertainty.
2. **Generic AI limitation:** Show a plausible but generic answer that lacks learning history and progress context.
3. **LCS auto-attachment:** Expose the current lesson, path progress, recent error, and relevant practice state that DevPath attaches automatically.
4. **Context-aware review:** Show an answer that identifies the learner's specific misconception and proposes a focused correction.
5. **Path correction:** Update the next task and briefly explain why it changed.

Diagnosis, search, architecture, and broader path views remain accessible, but they support the main loop instead of competing with it on the opening screen.

## Open Questions

- Which LCS fields should remain visible to the reviewer, and which should stay behind a concise summary?
- Should the final reviewer action point to the full learning path, the technical architecture, or a program-support call to action?
- Should architecture and search remain in secondary navigation or move to an appendix-style demo route?

## Success Criteria

- After a three-minute demo, a reviewer can explain LCS in one sentence without prompting.
- The reviewer can identify at least one material difference between the generic answer and the DevPath answer.
- The flow visibly connects the context-aware review to a changed next learning task.
- The primary story can be completed without visiting architecture or search pages.
- The flow remains understandable on both desktop and mobile layouts.
