## Vision

### 10x Check

Turn the prototype from a collection of independent feature simulators into one guided proof: DevPath understands why a learner failed, uses only context the learner approved, changes the quality of the AI review, adjusts the next learning task, and leaves evidence that supports employment preparation.

The three-minute experience should answer five reviewer questions without narration:

1. What failed?
2. What did generic AI miss?
3. What context did DevPath use, and did the learner control it?
4. What changed because of that context?
5. What useful artifact remains for employment preparation or mentor follow-up?

## Current State

- `DemoSection` stores diagnosis, path, sandbox, search, and review state centrally but renders them as independent pages.
- `_SandboxReviewSimulator` already provides failed code, execution output, and a static AI review.
- `_PathGeneratorSimulator` and `_PathExecutionBoard` already render a generated path and next tasks.
- `_ServiceDemoPage`, responsive grids, status semantics, and mobile navigation provide reusable presentation patterns.
- Existing tests verify independent feature navigation, sandbox review output, architecture content, and mobile section switching.
- LCS is documented as a thin context bridge that collects, sanitizes, assembles, and exposes learner-controlled snapshots. AI review and path correction remain consumers of that snapshot rather than responsibilities of LCS itself.

## Selected Implementation Approach

Use an explicit LCS-consuming learning-loop model rather than adding more conditional copy to the existing independent pages.

- `LearningLoopScenario` owns the static failed task, generic response, context snapshot, contextual review, path delta, employment evidence, and mentor summary fixture.
- `LearningLoopStage` defines valid progress through the guided flow.
- `LearningContextSnapshotView` contains learner-approved context fields plus source, freshness, and sanitized values.
- AI review and path correction consume the snapshot; LCS remains a context provider.
- Existing feature pages remain secondary exploration routes.
- New loop components should be extracted from `lib/main.dart` rather than extending the existing monolith.

## Review Decisions

### State Invalidation

When the learner navigates backward and changes an input, every derived result after that point is cleared. AI review, path delta, employment evidence, and mentor summary must be regenerated from the new approved context. The UI never presents an output produced from stale upstream state.

### Empty Context

Every LCS field is optional and can be disabled. When no fields are approved, the flow returns to the generic-answer state and disables contextual review, path correction, employment evidence, and mentor summary generation. The UI explains that these outputs require learner-approved context instead of silently producing equivalent-looking results.

### Employment Evidence

Employment-preparation evidence is an observable task-improvement record, not a hiring prediction, readiness percentage, or credential. It contains passed tests, the material before-and-after correction, demonstrated competency tags, and the next reinforcement task. Copy must not imply guaranteed employment or validated recruiter acceptance.

### Mentor Summary Privacy

The copied mentor summary contains only learner-approved context fields and results derived from those fields. Disabled field values and field names are omitted. The summary may state only that some learning context was excluded by learner choice.

### Restart Behavior

Restart is atomic. After the learner has progressed beyond the initial stage, restart requires confirmation, clears every selection and derived output together, returns focus to the guided-loop heading, and announces the reset through an accessibility live region. No partially reset state is valid.

### Responsive Guided Navigation

Desktop shows the full stage indicator. Mobile shows the current stage card, `current / total` progress, and one primary previous/next action; the complete stage list is available through an expandable control. Desktop and mobile consume the same `LearningLoopStage` state and transition rules.

### Error States

- `SummaryCopyFailure`: keep the summary visible, announce the failure, and offer retry plus manual text selection.
- `ScenarioDataMissing`: replace the loop body with a non-retryable fixture error panel and a route to existing feature exploration instead of asserting or rendering a blank area.
- `TransitionResult`: return `Applied`, `Rejected(reason)`, or `Failed(error)`. Duplicate completion is rejected silently, locked-stage access explains the prerequisite, and only invariant failure exposes error recovery.

Each error has a dedicated test. No catch-all or silent fallback is permitted.

### Accessibility Feedback

Every stage transition moves focus to the new stage heading. Stage changes, generated results, reset, copy success, and named failures are announced through scoped live regions without duplicating the full page. Keyboard order follows the visual order, all actions retain visible focus, and transition animation is disabled when reduced motion is requested.

## Architecture Review

### Learning Loop State Ownership

`LearningLoopState` is immutable. A pure transition function accepts the current state plus a named action and returns `Applied(nextState)`, `Rejected(reason)`, or `Failed(error)`. Any upstream input change clears all dependent downstream values in the same applied transition. Widgets render state and dispatch actions; they do not implement invalidation rules in local callbacks. No external state-management package is introduced.

```text
PrototypeHomePage
└── DemoSection
    ├── GuidedLearningLoop
    │   ├── LearningLoopState
    │   ├── transition(state, action)
    │   ├── LearningLoopScenario fixtures
    │   └── Stage views
    └── Existing feature exploration pages
```

### Module Boundary

The new feature lives under `lib/learning_loop/`:

- `models.dart`: immutable state, scenario, snapshot, evidence, and named result/error types.
- `reducer.dart`: actions, valid transitions, dependency invalidation, and invariant checks.
- `selectors.dart`: contextual review, path delta, evidence, visible summary, and clipboard payload derivation.
- `fixtures.dart`: static scenarios and generic/contextual response fixtures.
- `view.dart`: guided-loop composition and responsive stage navigation.
- `components.dart`: stage-specific presentation components shared by desktop and mobile.

`lib/theme/prototype_theme.dart` owns the application `ThemeData` plus a small semantic `ThemeExtension` for product-specific colors, typography, spacing, radii, focus, and motion values. It is the executable counterpart to `docs/DESIGN.md`; neither the shell nor `lib/learning_loop/` defines a second palette or feature-local design-token set.

`lib/main.dart` imports the shared theme and feature entry view and retains the existing shell and secondary exploration pages.

### Existing Page Isolation

The guided loop and secondary feature pages use the same typed scenario fixtures so learner, task, and expected outcome labels remain consistent. Interactive state is not synchronized: controls inside a page labeled `기능 탐색` cannot mutate or invalidate an in-progress guided loop. Returning to the loop restores its last valid state.

### Guided Session Lifetime

`_PrototypeHomePageState` owns the current immutable `LearningLoopState` and passes it with an update callback into `DemoSection`. Navigating to overview, architecture, or technology disposes the demo view but not the guided-session state. Returning reconstructs the view from the last valid state and offers `계속하기`. Existing diagnosis, path, sandbox, and search exploration state remains local and may reset independently.

### Derived Output Contract

Typed scenario fixtures contain source facts, not independently authored final cards. Pure selectors such as `contextualReview(state)`, `pathDelta(state)`, `evidence(state)`, and `mentorSummary(state)` derive every downstream result from the same learner-approved snapshot. Disabled fields cannot appear in a selector output, and empty context returns an explicit unavailable result.

### LCS Field Contract

`LearningContextSnapshotView` contains exactly four learner-controlled fields aligned with document 26: `currentContent`, `currentPath`, `activeTags`, and sanitized `recentErrors`. The first three default to included; `recentErrors` defaults to excluded and requires explicit opt-in. The current practice task, submitted code, and test outcome are inputs to the active interaction, not duplicated LCS fields.

### Clipboard Effect Boundary

`GuidedLearningLoop` receives an injectable `ClipboardWriter` callback. The production default delegates to `Clipboard.setData`; tests inject deterministic success, named failure, and delayed completion callbacks. The view dispatches `CopyRequested(attemptId)` before the effect and `CopySucceeded(attemptId)` or `CopyFailed(attemptId)` afterward, while the reducer remains platform-free and rejects stale attempt IDs.

## Error & Rescue Review

### Error Transport

Expected domain outcomes use sealed result types rather than exceptions. The reducer returns `Applied`, `Rejected`, or `Failed`; selectors return `Derived`, `Unavailable`, or a specific `Failure`. Only platform APIs throw across the feature boundary; `Clipboard.setData` `PlatformException` is caught and mapped to `SummaryCopyFailure`. Catch-all exception handling is prohibited.

### Clipboard Effect Concurrency

Each copy request receives an attempt ID. While the latest request is in flight, the copy action is disabled. Success or failure updates state only when its attempt ID is still current; restart invalidates the ID so stale platform completions cannot update the reset view.

### Fixture Validation

All scenarios are validated once when `_PrototypeHomePageState` initializes in debug and release builds. The root stores a `ScenarioCatalogResult` and creates `LearningLoopState` only from a successful catalog. Validation checks required non-empty source facts, valid stage references, at least one selectable context field, a material path delta, selector compatibility, and absence of contradictory evidence. Failure produces a non-retryable `ScenarioDataMissing` panel with a route to existing feature exploration; fixture tests fail CI so an invalid catalog cannot ship intentionally.

## Security Review

### Public Fixture Sanitization

Only pre-sanitized fixture values are allowed. The fixture validator and tests reject secret-like tokens, email addresses, absolute local paths, personal identifiers, and raw error-log values. Runtime masking is defense in depth only and is not accepted as a reason to commit sensitive source data.

### Clipboard Consent

The visible final-summary text and clipboard payload come from the same selector. The explicit action is labeled `현재 보이는 내용 복사`; no hidden metadata, disabled field, internal ID, or alternate serialization is appended. The new guided scenario uses a role/situation label rather than a personal name or pseudonym.

## Data Flow & Interaction Review

### Cross-Section Navigation

Leaving the demo section preserves the last valid guided-loop state in memory. Returning offers `계속하기` at that stage. Navigation does not persist state to local storage, URL parameters, or a server; page refresh starts a new session. Only explicit restart clears the in-memory loop during the page session.

### Stage Navigation Permissions

The learner may navigate to the current stage or any completed stage. Future stages are disabled and expose the unmet prerequisite in semantics/help text. Changing data in a completed stage invalidates all later stages and derived outputs before navigation can advance again.

### Idempotent Stage Completion

Every completion action carries its expected source stage. The reducer advances only when the expected stage equals the current stage. A repeated action from an already completed stage returns `Rejected(duplicateAction)` without user-facing error UI; locked-stage access returns `Rejected(prerequisiteMissing)` with prerequisite guidance. An impossible state invariant returns `Failed(invalidLoopState)` and preserves the last valid state.

### Static Freshness Semantics

Freshness uses scenario-relative labels such as `이번 실습 실행`, `현재 경로 기준`, and `진단 완료 시점`. The context area carries a visible and semantic `정적 샘플` label. It does not generate current timestamps or imply that live data was fetched.

## Code Quality Review

### Implementation Sequence

Implementation uses two behaviorally distinct commits on the same branch. Commit 1 is a behavior-preserving extraction of shared prototype widgets and must pass the existing test suite before any LCS behavior is added. Commit 2 adds the guided-loop domain, UI, effects, tests, documentation, and delivery changes. Structural moves and product behavior are never mixed in the same commit.

### Typed Identifiers

`ScenarioId`, `ContextFieldId`, and `LearningLoopStage` are typed identifiers. New feature code does not use user-facing strings as map keys and does not force-unwrap fixture lookups. Missing identifiers return `ScenarioDataMissing` through the sealed result contract.

### Shared Presentation Widgets

Reusable presentation primitives currently private to `main.dart`, including the panel, result row, responsive grid, and status treatment, move to `lib/widgets/prototype_components.dart` with public feature-neutral names. Existing pages and the guided loop import the same implementations. Theme construction and semantic tokens move separately to `lib/theme/prototype_theme.dart`; widgets read `ThemeData` and its semantic `ThemeExtension` instead of duplicating hexadecimal colors or feature-local spacing constants. Feature-specific stage widgets remain under `lib/learning_loop/`.

## Test Review

### Test Layers

- Unit: every reducer transition, invalid transition, dependency invalidation, fixture invariant, secret pattern, and selector result.
- Widget: guided start-to-finish flow, backward edits, empty context, locked future stages, double activation, restart confirmation, cross-section resume, copy success/failure/stale completion, keyboard focus, and live-region semantics.
- Diagnostics: inject a collecting `DiagnosticSink` and assert emitted typed metadata for each named failure; the event type has no LCS value, code, summary, or clipboard-payload field.
- Responsive/widget: boundary widths `599`, `600`, `979`, and `980` verify compact, medium, medium, and wide mode selection respectively. The comparison stage is the only medium layout allowed to use two columns.
- Accessibility/widget: text scale `1.0` and `2.0`, `44x44` minimum interactive targets, visible focus order, live-region announcements, and no clipped labels or horizontal overflow are asserted in widget tests.
- Golden: key stages at 390px, 768px, and 1280px using deterministic fixtures, fixed text scale cases, and reduced-motion state. Ubuntu is the canonical baseline platform; golden tests are tagged, baseline updates run only on Ubuntu/WSL, and CI compares every committed baseline.
- Font/design-system: theme-token mapping, approved palette literal deduplication, required font/license/hash presence, and Korean fixture glyph rendering are tested before guided-loop UI work can land.
- Release-browser QA: the generated release build is checked at browser zoom `200%` for retained content, reachable actions, no horizontal page scrolling, and no hover-only affordances. This is separate from Flutter text scale tests because browser zoom changes the rendered web viewport and CSS/device-pixel behavior.
- Existing regression: current independent demo pages, architecture content, navigation semantics, analyze, test, and release web build remain required.

### Privacy Combination Matrix

Selector tests enumerate all 16 combinations of the four LCS field selections. For every combination, disabled source values must be absent from contextual review, path delta, evidence, visible summary, and clipboard payload. The zero-field combination must return `Unavailable` for all contextual outputs, and the initial state must keep `recentErrors` excluded until explicit opt-in.

### Reducer Chaos Test

A deterministic fixed-seed test executes at least 1,000 mixed action sequences across advance, backward navigation, context toggles, restart, locked-stage activation, and copy completion events. After every action it asserts stage reachability, downstream invalidation, privacy exclusion, and copy-attempt invariants.

### Coverage Gate

CI runs `flutter test --coverage` and a dependency-free Dart checker that reads `coverage/lcov.info`, filters source files under `lib/learning_loop/`, and fails below 100% line coverage. Line coverage does not replace the explicit transition table, 16-combination privacy matrix, named failure tests, or user-flow tests.

### Mandatory Existing-Behavior Regressions

The behavior-preserving shared-widget extraction must add regression tests before feature work begins:

- 390px `_ServiceDemoPage` header renders without overflow and keeps the status below the subtitle.
- Demo navigation semantics expose the selected page and stable `페이지 열기` labels.
- Sandbox review updates its live status from waiting to complete without losing the API/storage/mode status pills.
- Existing overview, diagnosis, path, sandbox, search, architecture, and technology navigation remains reachable after public widget renames.

## Performance Review

The feature has no network, database, background job, unbounded list, or new package. Selectors operate over exactly four context fields and only the active stage is rendered. JavaScript and static assets have separate budgets so a font cannot bypass the existing gate. The current release baseline is `main.dart.js = 2,162,266 bytes`; CI warns above `2,270,379 bytes` (+5%) and fails above `2,378,493 bytes` (+10%). After the approved Pretendard files are selected, the same commit records `font_assets_bytes` and `total_web_assets_bytes` baselines from the release build. CI fails if a required font or license is absent, a pinned font hash changes without an explicit baseline update, font bytes exceed their recorded cap, or total web assets exceed +10%; total assets warn above +5%. A deliberate Flutter SDK or font revision may update only its affected baseline in a separately explained commit.

## Observability & Debuggability Review

Named failures emit a typed `DiagnosticEvent` through an injectable `DiagnosticSink`. The production sink delegates to `dart:developer.log`; tests inject a collector. The event type contains only stage, action, named error, and copy attempt ID, with no field capable of carrying LCS values, source code, error-log content, evidence text, mentor-summary text, personal identifiers, or clipboard payloads. Every failure remains user-visible with a recovery action; logs are diagnostic support, not a substitute for UI feedback.

The HTTP check complements widget tests; it does not claim to validate interaction behavior.

## Design & UX Review

### Design System Contract

Create `docs/DESIGN.md` before feature UI implementation and treat it as the visual source of truth for both the existing prototype shell and the guided loop. Preserve the established teal/neutral identity while defining semantic color roles, bundled Pretendard typography and fallbacks, type scale, spacing scale, border and radius rules, elevation policy, focus treatment, motion duration, reduced-motion behavior, and success/warning/error/inactive states. The contract must state when panels, cards, chips, stage indicators, comparison rows, and evidence blocks are appropriate so the new flow reuses a coherent vocabulary instead of adding feature-local styling constants. Font assets and their license must be repository-controlled, work without runtime font downloads, and remain within the approved bundle budget.

### First-Screen Hierarchy

The overview hero leads with the failed-practice problem, the limitation of a generic answer, and the promise of an AI mentor that knows the learner-approved context. `3분 데모 시작` is the single primary action. Broad platform capabilities, architecture, and technology remain available below or through secondary navigation rather than competing with the primary message.

The primary action switches to `데모 > 학습 루프`, which is the default demo destination and retains the existing top-level `데모` location indicator. The four independent simulators move behind a secondary `기능 탐색` entry inside the demo section. The guided loop always offers a clearly labeled route to `기능 탐색`, and returning from feature exploration restores the last valid guided-loop stage rather than starting a competing session.

The emotional arc begins with `막힌 이유 찾기`, not a learner-facing failure verdict. Opening copy separates the failed test from the learner's identity, acknowledges that a generic answer lacks the approved learning context, and offers one concrete next action. Each subsequent stage increases agency through visible context control and causal explanations. Completion restores control with observable improvement evidence, never confidence theater, deficit scoring, or fear-based employment claims.

The current four-tile `진단 · 경로 · 실습 · 리뷰` feature preview is removed from the hero. In its place, one evidence-led preview pairs a failed-test excerpt with a compact `일반 AI → DevPath AI` material difference and the learner-approved context responsible for that difference. This preview is asymmetric, left-aligned, and content-driven; it must not become an icon-card grid, decorative illustration, generic capability mosaic, or centered marketing composition.

### Generic vs Contextual Comparison

Desktop uses a two-column comparison with matching content order. Mobile stacks `일반 AI` before `DevPath AI`. Both layouts highlight the same material differences and preserve explicit labels; color alone never carries the distinction.

Responsive behavior uses three intentional modes: compact at `<=599px`, medium at `600-979px`, and wide at `>=980px`, aligned with the existing shell breakpoint. Compact uses a single column, current-stage card, `current / total` progress, and one dominant previous/next action. Medium keeps the shell's bottom navigation and the compact stage indicator, uses a single content column for every stage except the generic/contextual comparison, and allows that comparison alone to render as two matched columns when each side retains a readable measure. Wide uses the full stage indicator and two-column comparison. No mode merely scales up the compact layout or introduces a second session model.

Accessibility acceptance criteria are testable: every interactive target is at least `44x44` logical pixels; body text and essential iconography meet `4.5:1` contrast, while qualifying large text meets `3:1`; keyboard focus is always visible and follows visual order; dynamic stage, reset, copy, and error outcomes use deliberate live-region announcements without duplicate speech. At browser zoom `200%` and Flutter text scale `2.0`, the guided flow must retain all content and actions without horizontal page scrolling, clipped labels, overlapping controls, or hover-only affordances.

### Learning Path Delta

The path stage renders a focused diff rather than two complete boards. Unchanged stages are collapsed; the prior next task, revised next task, and learner-approved context that caused the change are explicit. The changed task is conveyed through text, iconography, and structure rather than color alone.

### Context Selection Controls

LCS fields render as checkbox cards containing the field label, sanitized sample value, source, scenario-relative freshness, and inclusion state. The full card is keyboard and pointer actionable, semantics expose selected state, and disabling a field immediately updates the approved-context preview.

If the learner returns to this stage and changes any approved field, the view remains on the context-selection stage while every dependent result is cleared atomically. An inline notice states `선택이 바뀌어 이후 결과가 초기화되었습니다`, identifies the affected outputs without exposing excluded values, and presents `새 결과 생성` as the single primary recovery action. No stale comparison, path delta, evidence, or summary remains visible behind the notice.

### Completion Hierarchy

The final stage presents observable employment-preparation evidence first, the mentor-summary preview second, and copy as the secondary action. Restart and broad feature exploration are tertiary actions. The completion screen does not present readiness scores, credentials, hiring predictions, or platform navigation at equal visual weight with the learner's result.

Employment-preparation evidence renders as a traceable record in this fixed order: verified test outcome, material before/after correction, competency tags each linked to the observed change that supports the claim, and the next reinforcement task. The record has no aggregate percentage, badge, rank, or employability label. The mentor summary is visually separate and subordinate so copying a handoff cannot be mistaken for issuing a credential.

### State Machine

```text
IDLE
  └─ start ─▶ 1 FAILURE + GENERIC AI
                 └─ complete(1) ─▶ 2 LCS PREVIEW
                                      ├─ empty ─▶ contextual outputs unavailable
                                      └─ complete(2) ─▶ 3 CONTEXTUAL REVIEW
                                                            └─ complete(3) ─▶ 4 PATH DELTA
                                                                                  └─ complete(4) ─▶ 5 EVIDENCE + SUMMARY

completed stage ── back/edit ──▶ that stage, all later outputs cleared
any progressed stage ── confirmed restart ──▶ IDLE
future stage ──▶ Rejected(prerequisiteMissing), state preserved
duplicate completion ──▶ Rejected(duplicateAction), state preserved
impossible invariant ──▶ Failed(invalidLoopState), state preserved
```
